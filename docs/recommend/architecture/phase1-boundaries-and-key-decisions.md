# 一期边界与关键决策

## 背景

当前目标不是全面替换老 PHP，而是先把“千人千面”快速做出来，同时确保今天的架构和工程标准能够直接复用到未来的全量 Go 迁移中。

## 一期硬约束

### 业务边界

- 老 PHP 项目一期 `不改业务代码`
- 老 PHP 继续承担：
  - 交易主流程
  - 后台管理
  - ERP
  - CRM
  - 历史接口
- Go 一期重点承担：
  - 首页推荐
  - 搜索重排
  - 类目重排
  - 商品详情相关推荐
  - 个性化活动/优惠券决策
  - 用户画像
  - 埋点与推荐归因

### 接入边界

- 不在 PHP 页面中插推荐组件
- 不依赖 PHP 新增接口或埋点
- 采用 `网关接管入口页/新页面/BFF` 的方式上线
- 老交易链路保留，推荐和展示层逐步切到 Go

### 实时性目标

- 画像更新目标：`1~5 分钟`
- 首页推荐、搜索重排优先做到近实时
- 其他触点允许短 TTL 缓存

### 基础设施硬决策

- 应用框架：`go-zero`
- 服务间协议：`gRPC`
- 对外接口：`HTTP/JSON + OpenAPI`
- 配置中心：`Nacos`
- 可观测性：`OpenTelemetry`
- 日志后端：`Loki`
- 指标后端：`Prometheus`
- 链路后端：`Jaeger` 或 `Tempo`
- 事件总线：`Kafka`
- 缓存：`Redis`
- 主持久化：`MySQL`
- 检索：`Elasticsearch/OpenSearch`
- 部署平台：`云托管 Kubernetes`

### 数据边界

- 老 PHP 仍是交易主写源
- Go 一期只做旁路读模型，不改老交易主表
- 通过 `CDC/binlog` 或过渡性增量同步获取数据

## 为什么这样定

### 为什么不改 PHP

- 当前最急的是千人千面上线
- 老系统耦合较深，直接改业务代码风险高
- 推荐系统可以先独立建设，收益更快体现

### 为什么不用 Redis 做主事件流

- 推荐系统要支持近实时、可回放、可扩展
- 曝光、点击、搜索、下单、支付等行为链路后续会持续放大
- Kafka 更适合做推荐和画像的数据平面

### 为什么从第一天就用 OpenTelemetry

- 避免后续统一埋点、日志字段、trace 上下文时返工
- 推荐系统尤其依赖：
  - 请求追踪
  - 实验归因
  - 推荐结果回放
  - 用户行为链路分析

### 为什么从第一天就用 Nacos

- 推荐系统会天然存在大量动态配置：
  - 策略阈值
  - 兜底开关
  - 流量开关
  - 实验参数
  - 运营白名单
- 这类配置不适合只靠静态文件维护

### 为什么一期就按云托管 K8s 设计

- 今天先搭一套适合未来全量迁移的底座，比以后整体翻修成本更低
- 推荐系统涉及多个服务与中间件，长期还是要进入标准云原生体系

## 一期上线方式

### 页面与入口策略

- 新入口或被网关接管的页面直接走 Go
- 推荐触点优先级：
  1. 首页推荐
  2. 搜索重排
  3. 类目重排
  4. 商品详情相关推荐
  5. 个性化活动/优惠券

### 流量策略

- 新流量优先进入 Go 页面
- 老页面继续由 PHP 提供
- 可以按用户、渠道、入口、地域做灰度

## 一期成功标准

- 推荐系统可独立部署、监控、回滚
- 画像更新可在分钟级生效
- 首页和搜索结果不再是统一排序
- 曝光、点击、下单、支付可以归因到实验和策略
- 所有基础设施标准可直接复用到后续全量 Go 迁移

## 一期不做的事

- 不做 PHP 全量替换
- 不做所有后台模块服务化
- 不做交易主写迁移
- 不做一开始就复杂的机器学习平台
- 不做一开始就完整数据中台

## 对后续迁移的直接帮助

一期完成后，后续全量 Go 迁移可以直接复用：

- 网关体系
- 鉴权与身份体系
- Nacos 配置体系
- OpenTelemetry 可观测性体系
- Kafka 事件体系
- Monorepo 工程结构
- 推荐与画像域能力
- 新页面/BFF 接管模式
