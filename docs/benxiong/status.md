# 项目状态字段说明
---

## 1. 通用约定

| 模式 | 含义（多处表共用） |
|------|---------------------|
| `is_del` | 常见：`1` 正常/未删，`2` 已删除（逻辑删） |
| `status`（商品/分类/仓库等） | 常见：`1` 启用/上架，`2` 停用/下架（具体以各模块查询条件为准） |
| `examine_status`（商品 `product`） | 后台与接口常用：`2` 审核通过可售；`1`/`3` 等见商品管理筛选 |

## 2. 订单主表 `order`

### 2.1 `order_status`（订单流程）

代码注释（商家端 `Shop` 模型）：

> 1-未支付 2-已支付 3-已发货 4-配送中 5-已完成 6-已取消 7-退款中 8-已退款 9-已送达待配送 10-骑手已送达 11-拒绝退款

| 值 | 含义（以后台 `Order` 列表与商家端展示为主；部分文案在不同控制器略有差异） |
|----|-----------------------------------------------------------------------------|
| 1 | 待支付 |
| 2 | 待发货（支付回调后常见状态） |
| 3 | 已发货（如 ERP「发货」后）；商家端也有「已发货，送往仓库中」等表述 |
| 4 | 配送中（分配骑手后常见） |
| 5 | 已完成 |
| 6 | 已取消 |
| 7 | 退款中 |
| 8 | 已退款 |
| 9 | 到仓/已送达待配送/已接单待配送（不同页面文案不一致，**数值均为 9**；常与入库、扫码流程相关） |
| 10 | 骑手已送达 |
| 11 | 拒绝退款 |

说明：后台 `application/admin/controller/Order.php` 列表里对 `order_status == 9` 写了两次不同中文（「到仓库」与「已送达待配送」），实际执行时**先匹配的分支生效**，属历史文案重复。

## 3. 订单商品行 `order_goods`

### 3.1 `order_status`

与主订单支付、发货流程联动；支付回调里将 `order_status` 从 `1` 更新为 `2`。统计「待发货销量」等场景常用 `order_status = 2`。

### 3.2 `refund_status`

ERP 扫码注释等处：

| 值 | 含义 |
|----|------|
| 1 | 无退款 / 正常 |
| 2 | 退款中 |
| 3 | 退款成功 |
| 4 | 退款失败 |

### 3.3 `storage_status`（行维度入库）

| 值 | 含义 |
|----|------|
| 1 | 未入库 |
| 2 | 已入库 |

### 3.4 `refund_type`（行维度）

业务里常见：`0` 与押金相关判断、`1` 退货退款类、`2` 押金等（见 `User` 退款申请逻辑）。

## 5. 骑手任务 `rider_order`

### 5.1 `type`

| 值 | 含义 |
|----|------|
| 1 | 商品配送订单 |
| 2 | 退货/售后取送关联 `order_refund_record` |
| 3 | 押金筐回收（关联 `egg_basket_return`） |

### 5.2 `status`（骑手端列表参数 `order_status` 同义）

接口注释：`待取货 1`、`待送达 2`、`已完成 3`。

| 值 | 含义 |
|----|------|
| 1 | 待取货（已派单、未完成取货确认） |
| 2 | 配送中/途中（如已「确定取货」） |
| 3 | 已完成（如已「送达」并上传照片等） |

注意：请求里 `order_status` 为 `0` 或字符串 `"0"` 时，PHP `empty()` 为真，**可能不加状态条件**（见 `Rider` 模型 `getOrders`）。
