## 1 退押金（鸡蛋筐 / 押金容器回收）测试说明

本文档面向测试同学，覆盖 **用户端申请 → 司机取回 → 后台财务退款** 的主链路及高风险分支，并给出 **建议优先级** 与 **通过标准**。接口路径以实际路由为准（一般为 `/api/Basket/...`，需带登录 `Token`）。

---

## 2. 前置条件（不满足则大部分用例无法执行）

| 条件 | 说明 |
|------|------|
| 测试账号 | 至少 2 个 **普通用户**（一个在白名单、一个不在），用于验证灰度（若已开启，见第 8 节）。 |
| 押金订单 | 存在 **已完成**（`order_status = 5`）、**已支付**（`pay_status = 2`）且订单明细含 **押金商品**（`product.is_deposit = 1`），且商品 **`deposit_type` 或类目** 能被系统识别为有效押金类型。 |
| 可退额度 | 该订单上仍有 **未占满** 的可退筐数（无进行中的冲突申请，或已取消/已结束单不占额度，以产品规则为准）。 |
| 司机 | 有可用 **司机账号** 能登录司机端，并能在任务列表看到 **退筐类** 任务（`rider_order.type = 3` 对应退筐，以实际端上展示为准）。 |
| 后台 | 有测试环境 **财务/运营** 账号能打开回收单并操作「财务退款」（以菜单名为准）。 |

---

## 3. 回收单状态（用户端展示名，便于对账）

主状态 `egg_basket_return.status`（与接口 `status_name` 对应关系以代码为准）：

| 值 | 名称（参考） |
|----|----------------|
| 1 | 待分配司机 |
| 2 | 待司机取回 |
| 3 | 已取回待财务退款 |
| 4 | 已退款 |
| 5 | 退款失败 |
| 6 | 已取消 |

财务侧另有 `finance_status`（如：待退款、已退款、退款失败等），后台列表若展示需一并核对。

---

## 4. 建议测试优先级

### P0（必须通过，阻塞发布）

1. **完整正向链路**：用户能查到可退订单 → 提交申请成功 → 司机任务生成 → 司机确认取回并填写 **实际筐数** → 后台可发起财务退款 → 回收单进入 **已退款**，金额与 **实际筐数 × 单价** 一致（或与后台展示一致）。
2. **申请校验**：可退数量 **不得超过** 订单剩余可退筐数；押金类型非法或商品非押金时应失败并提示合理。
3. **司机实际筐数**：实际筐数 **≤ 申请筐数**，且 **> 0** 才能进入待财务退款；财务退款使用 **`actual_refund_amount`**（由实际筐数计算），不得按申请筐数全额退（见第 6 节专项）。
4. **灰度（若配置开启）**：仅白名单用户可调用用户端退筐接口；非白名单一律被拒（见第 8 节）。

### P1（强烈建议）

5. **并发与重复**：同一订单连续两次申请、或列表刷新的幂等表现（无 duplicate 异常、提示清晰）。

### P2（时间允许）

6. 多押金类型、多行 `order_goods` 拆分明细时的金额汇总。
7. 微信支付 / 支付宝退款失败分支：回收单是否进入 **退款失败**、是否可再次发起财务操作（以当前后台能力为准）。

---

## 5. 推荐测试步骤（主链路拆解）

### 5.1 用户端

1. 调用 **可退订单列表**（如 `getEligibleBasketReturnOrders`）：  
   - **通过**：非空数据中订单含押金明细、`meta` 或字段合理；白名单外账号在灰度开启时被拒绝（预期）。  
2. 进入 **订单退筐信息**（如 `getOrderBasketReturnInfo`）：押金类型、最大筐数、联系人等字段完整。  
3. **创建申请**（如 `createBasketReturn`）：填写筐数、联系方式、地址等 → 返回 `return_no` / `return_id`，回收单状态为待分配或待司机取回（取决于是否自动匹到司机）。  
4. **回收单列表 / 详情**（如 `getBasketReturnList`、`getBasketReturnDetail`）：与库表或后台一致。

### 5.2 司机端

1. 在任务列表找到对应 **退筐** 任务。  
2. **确认取货 / 送达** 流程以你们 App 为准；关键是 **送达时填写实际取回筐数**。  
3. **通过**：回收单主表 `actual_basket_count`、`actual_refund_amount` 与填写一致；状态进入 **已取回待财务退款**（或等价状态）。

### 5.3 后台

1. 在回收单列表中找到该单，核对申请筐数、实际筐数、应退金额。  
2. 执行 **财务退款**：  
   - **通过**：状态 **已退款**，支付渠道返回成功或测试环境 Mock 成功；失败则进入 **退款失败** 且可追踪原因（若有字段）。

## 6. 怎样算「测试通过」（本轮）

同时满足：

1. **P0** 全部用例在当前测试环境 **执行通过**，无未关闭的 **致命 / 严重** 缺陷。  
2. 主链路 **至少 1 条** 从申请到财务退款 **端到端走通**（含司机填实际筐数、后台退款）。  
3. **金额**：展示金额、应退金额、实际退款字段 **一致且可追溯**（截图 + 单号记录）。  
4. 若开启灰度：**白名单 / 非白名单** 行为与配置一致。  
5. 已知限制（如仅支持微信或支付宝退款）已在测试报告中注明。

---

## 7. 灰度开关（若已配置）

配置项一般为 `basket_return_gray`（`application/config.php` 或环境变量）：

- **`enable` 为 false**：全员可使用退筐接口（与名单无关）。  
- **`enable` 为 true**：仅 **`uids`** 中的用户可访问用户端退筐相关接口；其他人应收到统一拒绝文案。

测试建议：用 **白名单用户** 与 **非白名单用户** 各测一轮列表与创建接口。
